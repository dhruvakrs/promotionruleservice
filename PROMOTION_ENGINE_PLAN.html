1. Overview & Vision

Product: PromoEngine

A cloud-native SaaS promotion calculation engine. Retailers and e-commerce platforms send a transaction (store, customer, items + quantities) and receive back calculated promotions, discounts, and final prices — in milliseconds. No legacy dependencies, no vendor lock-in, pure REST JSON API.

Target Market

Any retailer or e-commerce platform needing real-time promotion calculation at the point of sale or online checkout. Multi-tenant SaaS architecture.

Core Capability

Applies complex promotion rules (percentage discounts, amount off, buy-X-get-Y, coupons, bundles) across an entire transaction in a single API call.

Key Design Principle

Zero XML in the public API. Clean JSON in, clean JSON out. Rule definitions stay internal — callers never see engine internals.

What This Service Replaces (Internal Reference Only)

Internally this consolidates two legacy systems into one product. No external artefact or class name should reference those origins.

Legacy Core Service (Spring Boot)SOAP XML, OpenSearch + Oracle enrichmentMerged into PromoEngine, SOAP eliminated

Technology Stack

Spring Boot 3.2.2 Java 17 Maven — JAR packaging OpenSearch Oracle DB (fallback) AWS S3 SDK v2 SpringDoc OpenAPI 3 Lombok Promotion Rule Engine JAR (internal)

2. Promotion Rule Engine JAR — Analysis

The proprietary promotion rule engine ships as a JAR. It was fully decompiled and analysed. The key discovery changes the entire implementation strategy.

Critical Finding — Two API Layers Inside the JAR

Layer 1 (High-Level / XML): A simulator class that accepts an XML string and returns an XML string. This is what the legacy services use — requiring full JAXB marshalling/unmarshalling per transaction.

Layer 2 (Low-Level / Direct Java): A core engine interface that accepts typed Java objects (EngineItem) directly and delivers results via typed callbacks (PromotionResult). No XML required whatsoever.

By using Layer 2 directly, we eliminate all XML from the transaction path. No JAXB, no string parsing, no XML namespace manipulation.

Engine Adapter Pattern — Replacing the XML Layer

// Legacy approach (Layer 1) — XML in, XML out
legacySimulator.startSimulate(rpsdXmlString, 1);
String result = legacySimulator.getRpsd();   // XML string

// PromoEngine approach (Layer 2) — Java objects in, Java callbacks out — ZERO XML
engine.pushItem(invoiceLineItem);            // typed Java object
engine.pushItem(customerItem);
engine.commitItems();
engine.startCalculation();
// → onPromotionResult(PromotionResult result) called for each matched promotion

Legacy JAR API — Analysed, Not Used in New Service

The JAR was fully analysed to understand what it does. PromoEngine replaces the JAR entirely with Drools 8.x — see Section 2.5 below. The table below documents what was found during analysis.

JAR Replacement — Why Drools?

Why the JAR must be replaced (confirmed):
  • JNI / native C++ code — platform-specific binary; fails in Linux containers on ARM/x64 unless built for exact OS. Not cloud-native.
  • Black box — no debuggability, no test coverage possible, no way to inspect rule evaluation.
  • Vendor lock-in — MML (Marketing Markup Language) is a proprietary rule DSL. No open tooling, no community, no migration path.
  • Not containerisable — Fargate, EKS, Cloud Run all require fully portable JARs.
  • SaaS blocker — cannot redistribute a proprietary calculation engine as part of a market product.

Replacement: Drools 8.x (Apache)

ConcernLegacy JARDrools 8.x
RuntimeJNI / C++ native (platform-specific)Pure Java — runs anywhere JVM runs
Rule DSLMML (proprietary, undocumented)DRL (Drools Rule Language) — open standard
Thread safetyOne instance per concurrent request — requires pool of 20KieBase shared; StatelessKieSession per request — no pool needed
Zero-downtime reloadOne-at-a-time engine pool drainAtomicReference<KieBase> swap — truly instant, zero blocked requests
DebuggabilityNone — black boxFull rule tracing, Drools audit log, unit-testable rules
LicenseProprietaryApache 2.0 — redistributable as SaaS
CommunityNoneLarge, mature (Red Hat / JBoss project since 2001)

How Rules Are Stored and Compiled

// Rule stored in DB (rule_body column) as DRL text
// This replaces the proprietary MML (mmedCode) in MMED XML

rule "10% Off Electronics Over $100"
    salience 10
    when
        $tx : Transaction()
        accumulate(
            InvoiceItem(categoryCode == "ELEC", $amount : lineAmount),
            $total : sum($amount)
        )
        eval($total.doubleValue() > 100.0)
    then
        results.add(PromotionResult.builder()
            .promotionId("ELEC10PCT")
            .promotionType(0)           // 0 = STANDARD
            .description("10% off electronics over $100")
            .discountAmount($total.multiply(new java.math.BigDecimal("0.10")))
            .timesApplied(1.0)
            .build());
end

Compilation and Reload Flow

// PromotionRuleEngine — one per tenant
// KieBase = compiled ruleset; shared safely across concurrent sessions

void reload(List<RuleDefinition> rules) {
    KieServices ks = KieServices.Factory.get();
    KieFileSystem kfs = ks.newKieFileSystem();
    for (RuleDefinition r : rules) {
        kfs.write("src/main/resources/rules/" + r.getPromotionId() + ".drl", r.getRuleBody());
    }
    KieBuilder kb = ks.newKieBuilder(kfs).buildAll();
    if (kb.getResults().hasMessages(Message.Level.ERROR)) {
        throw new RuleCompilationException(kb.getResults().toString());
    }
    KieBase newBase = ks.newKieContainer(ks.getRepository().getDefaultReleaseId()).getKieBase();
    kieBaseRef.set(newBase);   // ← atomic swap — zero downtime, in-flight sessions unaffected
}

// Per calculation — StatelessKieSession is lightweight, created per request
CalculationResult calculate(EnrichedTransaction tx) {
    StatelessKieSession session = kieBaseRef.get().newStatelessKieSession();
    List<PromotionResult> results = new ArrayList<>();
    session.setGlobal("results", results);
    session.execute(EngineFactBuilder.build(tx));
    return new CalculationResult(results, tx.getSkippedItems());
}

3. Architecture

Single service, single responsibility: PromoEngine is one deployable unit. The promotion rule engine runs in-JVM — no internal HTTP hops. The enrichment layer (price, item properties, customer) is embedded. One container, one API, one deployment.

    │                                                             │
    │  session.execute(facts)                                     │
    │    → DRL rules fire, populate 'results' global             │
    │    → List<PromotionResult> (typed Java POJOs)              │

          │
          ▼  (no pool release needed — session is transient)

      promotionResults: List<PromotionResult>    ← Java POJOs, no XML
      lineResults:      Map<sku → LineResult>    ← per-line discount totals
      skippedItems:     List<String>
    }
          │
          ▼  PromotionResponseMapper.toResponse(result, request)
    CalculateResponse (JSON) ← rounded to 3dp, STANDARD/COUPON enum
          │
          ▼
    200 OK — application/json
Zero XML in the transaction path. The only XML used internally is the promotion rule definition file (loaded once at startup). This is the engine's internal rule language — not part of any API. External callers never see it.

8. Engine Layer Design (Drools 8.x)

No engine pool required. With Drools, KieBase is thread-safe and shared across all concurrent requests. A lightweight StatelessKieSession is created per calculation (cheap — no blocking queue, no acquisition timeout). Zero-downtime reload uses an AtomicReference<KieBase> swap.

EngineFactBuilder

// io.promoengine.engine.EngineFactBuilder
// Converts EnrichedTransaction → Drools facts. No proprietary classes, no XML.

class EngineFactBuilder {
    static List<Object> build(EnrichedTransaction tx) {
        List<Object> facts = new ArrayList<>();

        // Transaction header fact
        facts.add(Transaction.builder()
            .storeId(tx.getStoreId())
            .customerId(tx.getCustomerId())
            .transactionDate(tx.getTransactionDate())
            .build());

        // Customer fact (if data resolved from enrichment)
        if (tx.getCustomerData() != null) {
            facts.add(Customer.builder()
                .customerId(tx.getCustomerId())
                .customerGroupCode(tx.getCustomerData().getGroupCode())
                .build());
        }

        // One InvoiceItem fact per enriched line (items with no price are in skippedItems)
        for (EnrichedItem item : tx.getEnrichedItems()) {
            facts.add(InvoiceItem.builder()
                .sku(item.getSku())
                .storeCode(tx.getStoreId())
                .categoryCode(item.getCategoryCode())
                .subcategoryCode(item.getSubcategoryCode())
                .departmentCode(item.getDepartmentCode())
                .foodItem(item.isFoodItem())
                .unitPrice(item.getUnitPrice())
                .lineAmount(item.getUnitPrice().multiply(BigDecimal.valueOf(item.getQuantity())))
                .quantity(item.getQuantity())
                .build());
        }
        return facts;
    }
}

PromotionRuleEngine

// io.promoengine.engine.PromotionRuleEngine
// One per tenant. KieBase shared safely across concurrent StatelessKieSessions.

@Slf4j
class PromotionRuleEngine {
    private final String tenantId;
    private final AtomicReference<KieBase> kieBaseRef = new AtomicReference<>();

    /** Called at startup and on any rule change. Atomic — in-flight sessions complete normally. */
    void reload(List<RuleDefinition> rules) {
        KieServices ks = KieServices.Factory.get();
        KieFileSystem kfs = ks.newKieFileSystem();
        for (RuleDefinition r : rules) {
            kfs.write("src/main/resources/rules/" + r.getPromotionId() + ".drl", r.getRuleBody());
        }
        KieBuilder kb = ks.newKieBuilder(kfs).buildAll();
        if (kb.getResults().hasMessages(Message.Level.ERROR)) {
            throw new RuleCompilationException(tenantId, kb.getResults().toString());
        }
        KieBase newBase = ks.newKieContainer(
                ks.getRepository().getDefaultReleaseId()).getKieBase();
        kieBaseRef.set(newBase);   // atomic swap — zero downtime
        log.info("Rule engine reloaded for tenant={}, rules={}", tenantId, rules.size());
    }

    /** Thread-safe: StatelessKieSession created per call, discarded after. */
    CalculationResult calculate(EnrichedTransaction tx) {
        StatelessKieSession session = kieBaseRef.get().newStatelessKieSession();
        List<PromotionResult> results = new ArrayList<>();
        session.setGlobal("results", results);
        session.execute(EngineFactBuilder.build(tx));
        return new CalculationResult(results, tx.getSkippedItems());
    }
}

TenantEngineRegistry

// io.promoengine.engine.TenantEngineRegistry
// Holds one PromotionRuleEngine per tenant. Thread-safe.

@Component
class TenantEngineRegistry {
    private final ConcurrentHashMap<String, PromotionRuleEngine> registry =
            new ConcurrentHashMap<>();
    private final PromotionRuleService ruleService;

    /** Get (or lazily create and load) the engine for a tenant. */
    PromotionRuleEngine getEngine(String tenantId) {
        return registry.computeIfAbsent(tenantId, tid -> {
            PromotionRuleEngine engine = new PromotionRuleEngine(tid);
            engine.reload(ruleService.getActiveRules(tid));
            return engine;
        });
    }

    /** Trigger reload for one tenant (called after rule CRUD operations). */
    void reloadTenant(String tenantId) {
        getEngine(tenantId).reload(ruleService.getActiveRules(tenantId));
    }

    /** Reload all tenants (admin operation / startup resync). */
    void reloadAll() {
        registry.forEach((tid, engine) ->
                engine.reload(ruleService.getActiveRules(tid)));
    }
}

Parallel Enrichment (EnrichmentService)

// io.promoengine.enrichment.EnrichmentService
EnrichedTransaction enrich(CalculateRequest req) {
    List<String> skus = req.getItems().stream().map(OrderItem::getSku).toList();

    var pricesFuture    = CompletableFuture.supplyAsync(() -> priceRepository.findPrices(skus, req.getStoreId()));
    var propertiesFuture = CompletableFuture.supplyAsync(() -> itemRepository.findProperties(skus));
    var customerFuture  = CompletableFuture.supplyAsync(() -> customerRepository.findCustomer(req.getCustomerId()));

    CompletableFuture.allOf(pricesFuture, propertiesFuture, customerFuture).join();

    Map<String, PriceData>        prices     = pricesFuture.join();
    Map<String, ItemPropertyData> properties = propertiesFuture.join();
    Optional<CustomerData>        customer   = customerFuture.join();

    List<EnrichedItem> enrichedItems = new ArrayList<>();
    List<String>       skippedItems  = new ArrayList<>();

    for (OrderItem item : req.getItems()) {
        if (prices.containsKey(item.getSku())) {
            enrichedItems.add(EnrichedItem.from(item, prices.get(item.getSku()), properties.get(item.getSku())));
        } else {
            skippedItems.add(item.getSku());  // explicit — not silently dropped
        }
    }

    return EnrichedTransaction.builder()
        .storeId(req.getStoreId())
        .customerId(req.getCustomerId())
        .transactionDate(req.getTransactionDate())
        .enrichedItems(enrichedItems)
        .customerData(customer.orElse(null))
        .skippedItems(skippedItems)
        .build();
}

10. Promotion Rule Architecture

This section covers how promotion rules are defined, stored, loaded into the engine, and managed at runtime. This is a significant redesign from the legacy file-based approach.

What the Engine's Rule Format Contains

The calculation engine reads promotion rules from a structured XML format (the legacy system called this "MMED"). From decompiling the engine JAR, the full schema is:

<Mmed>
  <metaInformation>
    <name>Active Promotions v2.1</name>
    <description>Seasonal campaign — Q1 2024</description>
  </metaInformation>

  <!-- Presets: named groupings of items/customers used as rule conditions -->
  <preset>
    <definitionID>PRESET_ELECTRONICS</definitionID>
    <mmedGroup>1</mmedGroup>        <!-- 1=standard, 2=coupon -->
    <mmedCode>...MML rule code...</mmedCode>
  </preset>

  <!-- Promotions: actual discount/coupon rules -->
  <promotion>
    <definitionID>DEF_BUY2GET10</definitionID>
    <promotionID>PROMO_BUY2GET10</promotionID>
    <deletePromotion>false</deletePromotion>
    <priority>10</priority>
    <versionNumber>1.0</versionNumber>
    <promotionType>5</promotionType>    <!-- <10=STANDARD, >=10=COUPON -->
    <mmedGroup>1</mmedGroup>
    <startDate>2024-01-01</startDate>
    <endDate>2024-12-31</endDate>
    <mmedCode>...MML rule code defining conditions and outcomes...</mmedCode>
    <promotionText>
      <languageCd>EN</languageCd>
      <description>Buy 2 electronics — 10% off</description>
    </promotionText>
    <promotionText>
      <languageCd>TH</languageCd>
      <description>ซื้อ 2 ลด 10%</description>
    </promotionText>
  </promotion>
</Mmed>

The mmedCode field is the core of each rule — written in the engine's proprietary MML (Marketing Markup Language). It defines conditions (item group, quantity threshold, date range, customer segment) and outcomes (percentage off, fixed amount, free item, coupon). This MML syntax is owned by the engine and cannot be changed without replacing the engine itself.

Critical constraint: The engine's C++ native layer holds the parsed rules in-process. MmedType must reside in JVM heap during calculations — it cannot live in Redis, a message queue, or any external store during the calculation path. However, the source of truth and the management interface for rules can and should be fully modernised.

How Rules Are Loaded Into the Engine

Rules are read from the DB as promotion_rule records, each carrying a rule_body field containing DRL text. The PromotionRuleLoader compiles them into a KieBase and passes it to the PromotionRuleEngine for the tenant.

// PromotionRuleLoader — called at startup and on any rule status change

List<RuleDefinition> loadActiveRules(String tenantId) {
    return ruleRepository
        .findByTenantIdAndStatusAndDateRange(tenantId, "ACTIVE", LocalDate.now())
        .stream()
        .map(r -> RuleDefinition.builder()
            .promotionId(r.getPromotionId())
            .ruleBody(r.getRuleBody())     // DRL text stored in DB (replaces MML/mmedCode)
            .priority(r.getPriority())
            .build())
        .toList();
}

// Activating a rule — trigger engine reload for tenant
void activateRule(PromotionRuleEntity rule) {
    rule.setStatus("ACTIVE");
    rule.setActivatedAt(LocalDateTime.now());
    ruleRepository.save(rule);
    historyRepository.save(buildHistory(rule, "ACTIVATED"));
    tenantEngineRegistry.reloadTenant(rule.getTenantId());  // atomic KieBase swap
}

// Deactivating a rule — trigger engine reload for tenant
void deactivateRule(PromotionRuleEntity rule) {
    rule.setStatus("INACTIVE");
    ruleRepository.save(rule);
    historyRepository.save(buildHistory(rule, "DEACTIVATED"));
    tenantEngineRegistry.reloadTenant(rule.getTenantId());  // atomic KieBase swap
}

Zero-Downtime Reload — AtomicReference Swap

With Drools, there is no engine pool to drain. Reload is handled by building a new KieBase from the updated DB rules and atomically swapping the reference. In-flight StatelessKieSession instances complete normally with the old KieBase, which is then garbage-collected.

// PromotionRuleEngine.reload() — called by TenantEngineRegistry on any rule change
// In-flight calculations: continue with old KieBase (held in local variable in calculate())
// New calculations after swap: use new KieBase immediately
// Downtime: 0ms (no blocking, no pool drain, no queue wait)

void reload(List<RuleDefinition> rules) {
    KieBase newBase = compile(rules);     // build + validate DRL — may take 100–500ms
    kieBaseRef.set(newBase);             // atomic swap — never blocks callers
    log.info("[{}] Rules reloaded: {} active rules", tenantId, rules.size());
}

// calculate() always snapshots the reference first
CalculationResult calculate(EnrichedTransaction tx) {
    KieBase base = kieBaseRef.get();     // snapshot — reload-safe
    StatelessKieSession session = base.newStatelessKieSession();
    List<PromotionResult> results = new ArrayList<>();
    session.setGlobal("results", results);
    session.execute(EngineFactBuilder.build(tx));
    return new CalculationResult(results, tx.getSkippedItems());
    // session discarded — base reference released — eligible for GC if swapped
}
True zero downtime: Unlike the legacy pool-drain approach (19 engines serving, 1 reloading), Drools reload never blocks any calculation. All concurrent requests continue uninterrupted throughout the reload. Reload time = DRL compilation time only (~200ms for typical rule sets).

Bulk Import from S3 (migration / backwards compatibility)

The legacy MMED XML file upload endpoint is preserved as a one-time migration tool only. When called, PromoEngine parses the XML using a custom MmedXmlParser (JAXB — isolated to the import module only), extracts <promotion> elements, and for each one creates a promotion_rule DB row with a converted DRL body. A MmlToDrlConverter handles the translation of legacy MML expressions to equivalent DRL. Once imported to DB, the file is discarded and the DB is the sole source of truth.

JPA Entities

Promotion rules (source of truth): PromotionRuleEntity (promotion_rule table), PromotionPresetEntity (promotion_preset table), PromotionRuleHistoryEntity (promotion_rule_history table). 5

OpenSearch Document Models

PriceDocument (@Document(indexName="${promoengine.data.price-index}")), ItemDocument, CustomerDocument. Spring Data Elasticsearch.

6

Engine Layer — Most Critical (Drools)

POJOs: InvoiceItem, Transaction, Customer, PromotionResult (no JAR dependency).
EngineFactBuilder: converts EnrichedTransaction → Drools fact list. Zero proprietary classes.
PromotionRuleEngine: holds AtomicReference<KieBase>, exposes calculate() and reload().
TenantEngineRegistry: ConcurrentHashMap<tenantId, PromotionRuleEngine> — one engine per tenant.
RuleCompilationException: thrown when DRL text has errors (exposed to rule-CRUD caller).
Unit test: write a sample DRL rule, load it, run calculate(), assert PromotionResult fields.

7

Promotion Rule Management Layer

Rule loading: PromotionRuleLoader — reads active rules from DB → returns List<RuleDefinition> (each with DRL ruleBody). No XML, no MML.

Rule service: PromotionRuleService — full CRUD (create, activate, deactivate, rollback, dry-run test). On activate/deactivate: update DB, call tenantEngineRegistry.reloadTenant() → atomic KieBase swap.

DRL validation: on rule create/update, compile a test KieBase from the submitted DRL. Reject with 422 Unprocessable Entity if DRL has errors. Rules always validated before persisting.

Scheduled expiry: @Scheduled job runs nightly — finds rules where end_date < today AND status = 'ACTIVE' → deactivates and triggers reload.

Bulk import: S3RuleImporter (AWS SDK v2) + MmedXmlParser + MmlToDrlConverter — parses legacy MMED XML, converts MML → DRL, inserts as promotion_rule rows in DB, triggers tenant reload. S3 is import-only.

8

Enrichment Repositories

Price: OpenSearchPriceRepository (3-step fallback: store → sentinel 999999 → DB) + DatabasePriceRepository. Item: OpenSearchItemRepository + DatabaseItemRepository. Customer: OpenSearchCustomerRepository + DatabaseCustomerRepository.

9

EnrichmentService

Parallel CompletableFuture orchestrator. Assembles EnrichedTransaction. Items with no resolved price go into skippedItems list (explicit, not silent).

10

Request / Response DTOs

CalculateRequest + OrderItem with Bean Validation (@NotBlank, @NotEmpty, @Min(1)). CalculateResponse, OrderSummary, CalculatedItem, AppliedPromotion, PromotionResult, ErrorResponse.

11

PromotionResponseMapper

Maps CalculationResult + EnrichedTransaction → CalculateResponse. Logic: subtotal = Σ(line amounts), totalDiscount = Σ(promotion discounts after distribution), total = subtotal − totalDiscount. Per line: standardDiscount (promotionType<10), couponDiscount (≥10), finalAmount. Rounds all monetary values to 3 decimal places. Maps promotionType to "STANDARD" or "COUPON".

12

PromotionService

Main orchestrator. Resolves tenant from API key → calls EnrichmentService.enrich() → calls TenantEngineRegistry.getEngine(tenantId).calculate(tx) → calls PromotionResponseMapper.toResponse(). No pool acquisition or release — StatelessKieSession is created and discarded inside calculate(). Returns ResponseEntity with status 200 or 206 depending on skippedItems.
