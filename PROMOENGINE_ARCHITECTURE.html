Architecture & Implementation Blueprint  ·  rm3-engine-service (retired) MMED XML (legacy) → promoengine-service (new) PRS JSON (new format)  ·  March 2026

1. What Was Built

PromoEngine is a cloud-native, multi-tenant promotion calculation engine for retail SaaS. It replaces two legacy services and a proprietary engine JAR with a single Spring Boot 3.2 application using Drools 8.x as the open-source rule engine, a clean JSON API, and a portable rule file format (PRS) that eliminates vendor lock-in.

Core Engine

  • Drools 8.44.0.Final (Apache 2.0)
  • StatelessKieSession per request
  • AtomicReference zero-downtime reload
  • Multi-tenant engine registry
  • 142 Karate tests — all passing

Rule Formats

  • DRL stored in Oracle DB (CLOB)
  • MMED XML import (legacy, kept)
  • PRS JSON import (new, delta-aware)
  • 7 promotion type patterns
  • Batch import: single engine reload

API Surface

  • POST /calculate — 200/206
  • Full CRUD for rules + lifecycle
  • POST /import (MMED XML)
  • POST /import/prs (PRS JSON)
  • API key auth + multi-tenancy
Maven coordinates io.promoengine:promoengine-service:1.0.0-SNAPSHOT  ·  Root package: io.promoengine  ·  Context path: /promotionengine/api (override via PE_CONTEXT_PATH)

2. What Was Replaced & Why

❌ Retired (Legacy)

  • Proprietary engine JAR (RM3E-3.5.0.jar) — JNI/native C++, not containerisable
  • rm3-engine-service — Java 8 WAR, Tomcat 7, RPSD XML
  • Engine pool — ArrayBlockingQueue of 20, spin-loop + semaphore
  • MML DSL — proprietary Marketing Markup Language
  • MMED XML (new imports) — vendor-specific format
  • JAXB InvoiceRequest/Invoice — XML/SOAP in calculation path
  • Internal HTTP call — service-to-service for every calculation
  • Silent item drop — items with no price were silently skipped

✅ New Approach

  • Drools 8.x (Apache 2.0) — pure Java, container-native
  • StatelessKieSession — thread-safe, no pool needed
  • AtomicReference<KieBase> — zero-downtime hot reload
  • DRL (Drools Rule Language) — open standard, stored in Oracle DB
  • PRS JSON format — portable, delta-aware, vendor-free
  • Java POJOs — CalculateRequest/Response (JSON REST)
  • In-JVM calculation — zero internal HTTP hops
  • skippedItems[] — explicit in response, HTTP 206 Partial Content

4. Engine Architecture — 3-Tier Design

TIER 1 Source of Truth — Oracle Database

Table promotion_rule — one row per promotion. rule_body column is the complete DRL text (CLOB).

Table promotion_rule_history — every version of every rule, forever (no hard delete).

Rule lifecycle states: DRAFT → ACTIVE → INACTIVE

S3 is import-only — initial bulk load or migration. Database is always the source of truth.

PromotionRuleEntity PromotionRuleHistoryEntity PromotionRuleRepository PromotionRuleHistoryRepository
TIER 2 Compiled Rule Set — JVM Heap (per tenant)

PromotionRuleLoader reads ACTIVE rules from DB → List<RuleDefinition>

DRL text → KieFileSystem → KieBuilder → KieBase (compiled, immutable, thread-safe)

Reload: compile new KieBase → AtomicReference.set(newBase) — zero downtime. In-flight sessions complete with old KieBase.

PromotionRuleEngine PromotionRuleLoader RuleDefinition
TIER 3 Calculation — TenantEngineRegistry

ConcurrentHashMap<tenantId, PromotionRuleEngine> — one engine per tenant.

Per request: kieBase.newStatelessKieSession() → session.execute(facts) → session discarded.

No pool. No acquire/release. StatelessKieSession is cheap and disposable.

TenantEngineRegistry EngineFactBuilder CalculationResult 
// Zero-downtime reload pattern
void reload(List<RuleDefinition> rules) {
    KieBase newBase = compile(rules);     // ~100-500ms compile
    kieBaseRef.set(newBase);             // atomic swap — never blocks callers
}

// Thread-safe calculation — snapshot ref first
CalculationResult calculate(EnrichedTransaction tx) {
    KieBase base = kieBaseRef.get();     // snapshot — reload-safe
    StatelessKieSession session = base.newStatelessKieSession();
    List<PromotionResult> results = new ArrayList<>();
    session.setGlobal("results", results);
    session.execute(EngineFactBuilder.build(tx));
    return new CalculationResult(results, tx.getSkippedItems());
    // session auto-discarded — KieBase ref released
}

7. Rule Lifecycle

DRAFT
Created / updated
Not in engine
→ activate()
ACTIVE
Loaded in KieBase
Applied to calculations
→ deactivate()
INACTIVE
Removed from engine
Preserved in DB

Rules are never hard-deleted. Every state change is recorded in promotion_rule_history. Rollback to any previous version is supported via POST /api/v1/rules/{id}/rollback?version=N.

DRL validation before activation activate() compiles a test KieBase before changing status to ACTIVE. Invalid DRL throws HTTP 422 with errorCode=14. The rule remains DRAFT.

Batch operations (activateBatch / deactivateBatch) process N rules but trigger only ONE engine reload at the end — critical for bulk imports of thousands of rules.

8. MMED XML Format (Legacy — Kept for Backward Compatibility)

Still supported The MMED XML import endpoint (POST /api/v1/rules/import) is kept for backward compatibility with existing rule generator systems. No new rules should be authored in MMED format — use PRS JSON instead.

8.1 File Format — Real QA Format

<!-- Root element with namespace -->
<ns3:mmed xmlns:ns3="http://www.capgemini.com/rm3/common/mmed">
  <ns3:promotion>
    <promotionID>100556820</promotionID>
    <promotionType>3</promotionType>
    <priority>40</priority>
    <startDate>2023-01-06</startDate>
    <endDate>2043-12-31</endDate>
    <mmedCode>H4sIAAAAA...</mmedCode>  <!-- Base64+gzip+JavaSerial -->
  </ns3:promotion>
</ns3:mmed>

8.2 mmedCode Decoding

1

Base64 decode

Pad to multiple-of-4 if needed: int rem = s.length() % 4; if (rem != 0) s += "=".repeat(4-rem);

2

GZip decompress

GZIPInputStream → ByteArrayOutputStream

3

Strip Java serialization header

Magic: 0xAC 0xED 0x00 0x05 0x74 + 2-byte string length. Skip 7 bytes → read as ISO-8859-1.

9. PRS JSON Format (New — Recommended)

PRS (PromoRuleSet) is a portable, human-readable JSON format that replaces MMED XML for new rule authoring workflows. It covers all 7 real-world promotion scenarios, supports delta imports (upsert/delete per rule with version-based conflict detection), and has no vendor dependencies.

9.1 Full PRS JSON Example

{
  "format":      "prs",
  "version":     "1.1",
  "generatedAt": "2026-03-04T10:00:00Z",
  "tenantId":    "default",          // optional — overrides API key tenant
  "rules": [
    {
      "id":          "100556820",
      "version":     3,               // skip if DB version >= 3
      "action":      "upsert",         // "upsert" | "delete"
      "description": "Buy 10 electronics — 20 baht off",
      "promotionType": 3,
      "priority":    40,
      "startDate":   "2023-01-06",
      "endDate":     "2043-12-31",
      "trigger": {
        "type": "BUY_QTY",
        "groups": [{
          "skus":   ["824216", "108389"],
          "minQty": 10
        }]
      },
      "application": { "type": "MOST_EXPENSIVE", "count": 1 },
      "discount":    { "type": "FIXED", "amount": 20.00 }
    }
  ]
}

10. DRL Patterns — Generated by PrsDrlConverter

10.1 Standard DRL Header (prepended automatically by PromotionRuleEngine)

package io.promoengine.rules;
import io.promoengine.engine.model.InvoiceItem;
import io.promoengine.engine.model.Transaction;
import io.promoengine.engine.model.Customer;
import io.promoengine.engine.model.PromotionResult;
import java.math.BigDecimal;
import java.util.List;
global List<PromotionResult> results;

10.2 BUY_QTY + TRANSACTION — Buy N items, fixed baht off

rule "100556820"
  salience 40
  when
    $tx : Transaction(
        !transactionDate.isBefore(java.time.LocalDate.parse("2023-01-06")),
        !transactionDate.isAfter(java.time.LocalDate.parse("2043-12-31")))
    accumulate(
        io.promoengine.engine.model.InvoiceItem(sku in ("824216", "108389"), $q1 : quantity),
        $qty1 : sum($q1)
    )
    eval($qty1.intValue() >= 10)
  then
    results.add(PromotionResult.builder()
      .promotionId("100556820").promotionType(3)
      .description("Buy 10 electronics — 20 baht off")
      .discountAmount(new java.math.BigDecimal("20.0000")).timesApplied(1.0).build());
end

10.3 BUY_QTY + MOST_EXPENSIVE — Apply discount to most expensive item

    // ... qty accumulate conditions above ...
    $topItem : io.promoengine.engine.model.InvoiceItem(sku in ("824216", "108389"))
    not io.promoengine.engine.model.InvoiceItem(sku in ("824216", "108389"), unitPrice > $topItem.unitPrice)

10.4 SPEND_MIN — Spend minimum amount trigger

    accumulate(
        io.promoengine.engine.model.InvoiceItem(sku in ("111", "222"), $la1 : lineAmount),
        $amt1 : sum($la1)
    )
    eval($amt1.doubleValue() >= 185.0)

10.5 CROSS_GROUP — Multiple independent groups (AND logic)

    // Group 1: spend min on product A
    accumulate(
        io.promoengine.engine.model.InvoiceItem(sku == "SKU_A", $la1 : lineAmount),
        $amt1 : sum($la1)
    )
    eval($amt1.doubleValue() >= 500.0)
    // Group 2: buy min qty of product B
    accumulate(
        io.promoengine.engine.model.InvoiceItem(sku == "SKU_B", $q2 : quantity),
        $qty2 : sum($q2)
    )
    eval($qty2.intValue() >= 3)
