rm3-promotion-service

Merged Architecture Blueprint  ·  rm3-engine-service (legacy) + rm3-calculate-invoice-core-service → rm3-promotion-service (new)  ·  Feb 2026

1. Overview

The current promotion pipeline is split across two separate services that communicate over HTTP. This blueprint describes merging them into a single Spring Boot 3.2 service that embeds the RM3E engine directly, eliminates the internal HTTP hop, and provides a clean XML/SOAP API that is backward-compatible with the existing caller.

rm3-engine-service (dead-end)

  • Java 8 Servlet WAR, Tomcat 7
  • Accepts raw RPSD XML (engine format)
  • Thin wrapper — no business logic
  • Engine pool: spin-loop + semaphore
  • HTTP-only, no auth

rm3-calculate-invoice-core-service

  • Spring Boot 3.1.8, Java 17
  • Accepts SOAP XML InvoiceRequest
  • Enriches: price + item props + customer
  • Builds RPSD XML, calls engine via HTTP
  • Returns SOAP XML Invoice response

rm3-promotion-service (target)

  • Spring Boot 3.2.x, Java 17, JAR
  • Same SOAP XML interface (backward compat)
  • Engine pool embedded (ArrayBlockingQueue)
  • Zero internal HTTP hops
  • Oracle DB + OpenSearch + AWS S3

2.2 Core Service — What It Actually Does

The core service is not a simple proxy. It performs significant orchestration on every request:

#StepDetailFallback
1 Parse SOAP XML request Unwrap <x:Body> or <SOAP-ENV:Body> → unmarshal InvoiceRequest via JAXB Throw 400
2 Price lookup (parallel) OpenSearch index rm3getprice by store+articles. If missing, retry with sentinel store 999999 Oracle DB (ItemStoreValue table)
3 Item property lookup (parallel) OpenSearch index rm3product by article codes → returns up to 5 property pairs (pt1/v1 … pt5/v5) Oracle DB (ItemPropertyValues table)
4 Customer property lookup If Cus != 0: OpenSearch index rm3customerlist by customer number Oracle DB (CustomerPropertyValue table)
5 Build RPSD XML (ServletRequest) Map InvoiceRequest + enrichment data → JAXB marshal to SOAP-wrapped RPSD XML
Articles not found in price map are silently dropped		 —

2.3 Engine Service — Internal Pool

// Legacy: three separate maps — can desync
TreeMap<Long, RM3S> rm3sList           // engine instances
TreeMap<Long, Long> rm3sStatusList      // 0=idle, 1=busy, 5=loading
TreeMap<Long, Semaphore> rm3sSemList   // per-engine semaphore

// Acquire loop: 300 x 10ms sleep = 3 second timeout
for (i=0; i<300 && !found; i++) {
    for (engineSeq=0; engineSeq < maxEngines && !found;) {
        if (status==0 && semaphore.permits > 0) {
            semaphore.acquire(); status=1; found=true;
        }
    }
    if (!found) Thread.sleep(10);
}
Critical Engine Bug The engine releases the semaphore before resetting status to 0 (rm3sSemList.get(seq).release() then rm3sStatusList.put(seq, 0L)). There is no finally block — an exception thrown inside startSimulate() leaves the engine permanently busy with the semaphore released but status still set to 1. Over time, engines leak from the pool.

3. Problems & Gaps in Current Architecture

CategoryProblemImpact
Performance Internal HTTP hop between core and engine service adds network latency + serialization cost on every request Every invoice calculation pays XML marshal → HTTP → XML unmarshal twice extra
Reliability Engine pool leak: exception in startSimulate() corrupts pool state (semaphore released, status not reset). No finally block. Under load, engines leak. Service degrades silently until all 20 engines are "stuck".
Reliability CPU spin-loop: 300x Thread.sleep(10) wastes CPU under contention instead of blocking efficiently High CPU usage when all engines busy; thread starvation risk
Complexity RPSD XML is an internal engine format exposed as an API boundary. Core service must know the full RPSD schema just to call the engine. Any RPSD schema change requires changes in two services
Complexity Two separate deployment units for what is logically one operation Two Dockerfiles, two CI pipelines, two sets of env vars, two health checks to manage
Complexity SOAP XML in, SOAP XML out — no JSON API. All callers must implement SOAP parsing. Poor developer experience; testing requires XML tooling
Maintainability Engine service uses Java 8 + Tomcat 7 WAR — no modern Spring Boot features, no Actuator, no Swagger Cannot use Spring Boot ecosystem; requires separate web.xml config
Feature Gap No API key security on either service. Anyone who can reach the endpoint can call it. Security risk in production
Feature Gap Shadow engine routing is implemented in core service but not cleanly abstracted — hardcoded Random comparison in service layer Difficult to test or adjust without code changes

4.1 What Is Removed vs Kept vs New

Removed (from both legacy services)

  • rm3-engine-service as a separate process
  • Internal HTTP call (RestTemplate to SERVLET2_URL)
  • SOAP/RPSD XML as external API format
  • Java 8 / Tomcat 7 / WAR packaging
  • TreeMap + Semaphore spin-loop pool
  • web.xml configuration
  • System.out.println logging
  • Comparator service (not in scope)

Kept (business logic from core service)

  • OpenSearch → DB fallback pattern for all lookups
  • Mandatory item properties 888 (store) + 887 (USE_STORES)
  • Price lookup: regularPrice fetched internally from OpenSearch (not from client)
  • Price calculation: lineTotal = regularPrice × quantity
  • Items missing from price catalog are reported in skippedSkus[] (no silent drop)
  • Discount calculation logic (subtotal, totalDiscount, total, line-level amounts)
  • Promotion type discrimination (<10 = STANDARD, ≥10 = COUPON)
  • Sentinel store 999999 fallback in price lookup
  • MMED upload/reload/S3 download logic
  • Internal RPSD XML models (used only inside the service, never exposed)

New (in merged service)

  • REST JSON API — industry-standard, POS & web friendly (replaces SOAP XML entirely)
  • Human-readable field names: storeId, sku, quantity, subtotal, totalDiscount
  • Promotion type as string enum: "STANDARD" / "COUPON" (replaces cryptic SPI/CPI)
  • skippedSkus[] field — explicit reporting of articles not in catalog
  • ArrayBlockingQueue engine pool (no spin-loop)
  • ReentrantReadWriteLock for zero-downtime reload
  • Engine always returned in finally block (no leak)
  • Spring Boot Actuator health endpoint
  • Swagger / OpenAPI documentation
  • X-API-Key authentication filter
  • Single Dockerfile / single CI pipeline

Optional / Future

  • Shadow engine routing (feature-flag based)
  • JSON API endpoint alongside SOAP for new callers
  • Comparator service integration
  • Distributed tracing (Micrometer/Zipkin)
  • Response caching for repeated identical invoices

5. Data Flow

5.1 Full Request Processing Flow

── Inbound ──────────────────────────────────────────────────────────────
Client (POS terminal / website / API consumer)
  POST /api/v1/promotions/calculate
  Content-Type: application/json
  X-API-Key: <key>
  Body: JSON { storeId, customerId, transactionDate, items:[{sku, quantity}] }

── PromotionController ───────────────────────────────────────────────
  1. @Valid deserialise CalculateRequest (Jackson)
  2. generate requestId (UUID) if not provided by client
  3. delegate → PromotionService.calculate(request, requestId)

── PromotionService ───────────────────────────────────────────────────
  4. allSkus = request.getItems().map(sku) → Set<String>

  5. (CompletableFuture.supplyAsync) PriceDataService.getPrices(storeId, allSkus)
       a. OpenSearch rm3getprice by loc + articles
       b. if missing → retry with loc=999999
       c. if still missing → Oracle DB (ItemStoreValue)
     → Map<String, Double> skuRegularPriceMap

  6. (CompletableFuture.supplyAsync) ItemPropertyService.getProperties(allSkus)
       a. OpenSearch rm3product by SKU codes → SourceES (pt1-5, v1-5)
       b. if missing → Oracle DB (ItemPropertyValues)
     → Map<String, Product> itemPropertyMap

  7. (if request.getCustomerId() != null && != 0) CustomerDataService.getCustomer(cusNo)
       a. OpenSearch rm3customerlist by cusNo
       b. if missing → Oracle DB (CustomerPropertyValue)
     → List<CustomerList> customerProperties

  8. skippedSkus = allSkus - skuRegularPriceMap.keySet()  ← explicit, not silent

  9. wait all futures

── RpsdXmlBuilder (internal — never exposed to clients) ──────────────
  10. Build internal RPSD/Invoice JAXB model (engine's proprietary format):
     invoice.id         = requestId (or UUID)
     invoice.sreKey     = request.getStoreId()
     invoice.invoiceDate = request.getTransactionDate()
     invoice.custNo     = 0
     invoice.invoiceTime = 0
     invoice.valueInd   = 1
     invoice.customerProperty = [PropertyType{id, value}] (from step 7)

     for each item where sku in skuRegularPriceMap:
       invoiceLine.seqNo      = i++
       invoiceLine.itemType   = "A"
       invoiceLine.itemNo     = item.sku
       invoiceLine.quantity   = item.quantity
       invoiceLine.amount     = regularPrice × item.quantity
       invoiceLine.rm3EItemType = 4
       invoiceLine.properties += [pt1/v1 … pt5/v5]  (from step 6)
       invoiceLine.properties += {id:888, value:storeId}   ← mandatory
       invoiceLine.properties += {id:887, value:"USE_STORES"}  ← mandatory

     rpsd.hasMPXInvoiceDependency = true
     JAXB marshal → RPSD XML string  ← engine's internal format only

── EnginePool ─────────────────────────────────────────────────────────
  11. reloadLock.readLock().lock()                ← parallel simulate OK
  12. engine = queue.poll(3000ms)
      if null → throw EngineTimeoutException (HTTP 503)
  13. try {
        engine.rm3s.setDebugLevel(config)
        engine.rm3s.setNbrDecimals(config)
        engine.rm3s.startSimulate(rpsdXml, 1)
        rpsdXmlResponse = engine.rm3s.getRpsd()
      } finally {
        queue.offer(engine)                       ← ALWAYS returned
        reloadLock.readLock().unlock()
      }

── Response transformation → clean JSON (never XML) ──────────────────
  14. JAXB unmarshal rpsdXmlResponse → Rpsd (internal model)

  15. ResponseMapper.toCalculateResponse(rpsd, request, skippedSkus):
      → summary.subtotal   = Σ(line.amount)
      → summary.totalDiscount = Σ(invoiceResult.totalDiscountAmountAfterDistribution)
      → summary.total      = subtotal - totalDiscount
      → for each InvoiceResultType:
           PromotionResult{
             promotionId   = promotionNo
             type          = promotionType < 10 ? "STANDARD" : "COUPON"
             description   = promotionText
             timesApplied  = times
             discountAmount = totalDiscountAmountAfterDistribution
             prerequisites  = Pre[]{type, item}
           }
      → for each InvoiceLineType:
           CalculatedItem{
             sku            = itemNo
             quantity       = quantity
             unitPrice      = round(amount/quantity, 3)
             subtotal       = round(amount, 3)
             standardDiscount = Σ(lineResult.resultValue where promotionType<10)
             couponDiscount  = Σ(lineResult.resultValue where promotionType≥10)
             finalAmount     = subtotal - standardDiscount - couponDiscount
             appliedPromotions = [{promotionId, type, discountAmount}]
           }
      → round all monetary values to 3dp

  16. return ResponseEntity.ok(CalculateResponse{
        requestId, storeId, calculatedAt,
        summary, items, promotions, skippedSkus
      })

5.2 MMED Reload Flow

POST /api/v1/mmed/reload  (X-API-Key required)

MmedController → MmedService.reloadMmed()
  1. S3MmedDownloader.downloadToFile(bucket, key, localPath)
  2. mmed = RM3Interface.RM3CreateMmedTypeRefFromXML(localPath)
  3. for each extraPath in config:
        RM3Interface.RM3AppendMmedTypeRefFromXML(extraPath, mmed)
  4. enginePool.reloadAll(mmed):
       writeLock.lock()                          ← blocks all new simulate calls
       drain all engines from queue (poll up to 10s each)
       for each engine:
           RM3S fresh = new RM3S()
           result = fresh.checkMmed(mmed, debugLevel)
           if result != 0 → throw EngineInitException(41)
           engine.rm3s = fresh                   ← atomic swap
       return all engines to queue
       writeLock.unlock()                        ← simulate calls resume
  5. return MmedOperationResponse{success, reloadedAt}

6. Request/Response Contracts — REST JSON API

Design reference API contract modelled after Microsoft Dynamics 365 Commerce CalculateSalesDocument API, Shopify Cart API, and Stripe's error format — all of which power large-scale POS and e-commerce platforms globally. Core principle: the client sends only what it knows (SKU + quantity + store); everything else (price, item properties, customer segments) is fetched and enriched server-side.

7. Data Enrichment Layer

7.1 Price Lookup Strategy

PriceDataService.getPrices(store, articles):
  // Step 1 — OpenSearch with actual store
  BoolQuery: must(loc=store) + must(articles IN ids) + mustNot(cusListFilter exists)
  → collect found records; track remaining = articles - found

  // Step 2 — OpenSearch with sentinel store 999999
  if remaining not empty:
    BoolQuery: must(loc="999999") + must(remaining articles)
    → collect; update remaining

  // Step 3 — Oracle DB fallback
  if remaining not empty:
    SELECT from ItemStoreValue WHERE item IN remaining AND store = store
    → PriceRecord{article, regularPrice}

  return Map<articleCode, regularPrice>
Important business rule Articles not found after all three steps are silently omitted from the engine request. The client receives a response with fewer lines than requested. No error is returned. This behavior must be preserved.

7.2 Item Property Lookup Strategy

ItemPropertyService.getProperties(articles):
  // OpenSearch rm3product index
  BoolQuery: must(productCode IN articles)
  → map SourceES → Product{productCode, productDetail{pt1-5, v1-5}}

  // DB fallback (per missing article)
  if products empty or partial:
    SELECT property_type, value FROM ItemPropertyValues WHERE item IN missing
    → group by item → SourceES with up to 5 pt/v pairs

  return Map<articleCode, Product>

7.3 Customer Property Lookup Strategy

CustomerDataService.getCustomer(cusNo):
  if cusNo == 0: return empty list

  // OpenSearch rm3customerlist index
  BoolQuery: must(cusNo = cusNo)
  → List<CustomerList{cusList, value}>

  // DB fallback
  if IOException:
    SELECT from QS_CUS_CUS_PROPERTY_VALUES WHERE customer_number = cusNo
    → CustomerList{cusList=propertyType, value=...}

  These map directly to invoice.customerProperty[]{id=cusList, value=value}

8. Engine Pool Design

8.1 New Pool Architecture

@Component
public class EnginePool {
    private final ArrayBlockingQueue<EngineWrapper> pool;   // fair=true
    private final List<EngineWrapper>          allEngines; // for reload
    private final ReentrantReadWriteLock      reloadLock;
    private final AppProperties               config;

    // ── SIMULATE (read lock, parallel) ──────────────────────────────
    public <T> T execute(Function<EngineWrapper, T> task) {
        reloadLock.readLock().lock();
        try {
            EngineWrapper engine = pool.poll(config.getAcquireTimeoutMs(), MILLISECONDS);
            if (engine == null) throw new EngineTimeoutException(32);
            try {
                return task.apply(engine);
            } finally {
                pool.offer(engine);           // ALWAYS returned, even on exception
            }
        } finally {
            reloadLock.readLock().unlock();
        }
    }

    // ── RELOAD (write lock, exclusive) ──────────────────────────────
    public void reloadAll(MmedType newMmed) {
        reloadLock.writeLock().lock();        // blocks ALL new simulate calls
        try {
            // drain pool (wait for in-flight calls to finish)
            List<EngineWrapper> drained = new ArrayList<>();
            for (EngineWrapper e : allEngines)
                drained.add(pool.poll(10, SECONDS)); // wait for each engine
            // reinitialize each engine with new MMED
            for (EngineWrapper e : drained) {
                e.reloadMmed(newMmed, config);
                pool.offer(e);
            }
        } finally {
            reloadLock.writeLock().unlock();  // simulate calls resume
        }
    }
}

public class EngineWrapper {
    private volatile RM3S rm3s;

    public String simulate(String rpsdXml, AppProperties cfg) {
        rm3s.setDebugLevel(cfg.getDebugLevel());
        rm3s.setNbrDecimals(cfg.getNbrDecimals());
        rm3s.startSimulate(rpsdXml, 1);         // 1 = inline XML
        return rm3s.getRpsd();
    }

    public void reloadMmed(MmedType newMmed, AppProperties cfg) {
        RM3S fresh = new RM3S();
        long rc = fresh.checkMmed(newMmed, cfg.getDebugLevel());
        if (rc != 0) throw new EngineInitException(41, rc);
        this.rm3s = fresh;                       // volatile = atomic swap
    }
}

10. Layer-by-Layer Design

10.1 Configuration Layer

config/ AppProperties EnginePoolConfig OpenSearchConfig AsyncConfig SecurityConfig

AppProperties — @ConfigurationProperties("rm3") binds all engine, MMED, security, and S3 settings from application.yml.

EnginePoolConfig — @Bean that runs at startup: downloads MMED from S3, loads MmedType, creates N EngineWrapper instances with checkMmed(), populates EnginePool queue.

OpenSearchConfig — creates RestHighLevelClient from URI/credentials (same pattern as core service).

AsyncConfig — configures async thread pool for OpenSearch queries (parallel price + item property lookups).

10.2 Security Layer

security/ ApiKeyFilter 
// Paths skipped: /actuator/**, /swagger-ui/**, /v3/api-docs/**
// Checked: X-API-Key header == rm3.security.api-key property
// On fail: 401 + { success:false, errorCode:401, errorMessage:"Unauthorized" }
Note on /v1/invoice The primary invoice endpoint should also require X-API-Key. Existing callers (api-service) must be updated to pass the key. This is the only breaking change from the current no-auth setup.

10.3 Enrichment Layer

enrichment/ PriceDataService ItemPropertyService CustomerDataService

These three services replace the inline data-fetching code in ServletService.calculateInvoice(). Each implements the OpenSearch-first, DB-fallback pattern with proper CompletableFuture composition.

Key design decision: PriceDataService and ItemPropertyService are called in parallel via CompletableFuture.supplyAsync(). CustomerDataService is called sequentially only when cus != 0, to avoid unnecessary DB lookups for anonymous customers.

10.4 Service Layer — InvoiceService

service/ PromotionService

The central orchestrator. Replaces ServletService.calculateInvoice() from the core service. Key responsibilities:

  • Accept CalculateRequest JSON DTO (not SOAP XML)
  • Coordinate parallel data enrichment (price + item props + customer)
  • Track skippedSkus explicitly (SKUs with no price found)
  • Build RPSD XML internally via JAXBUtil (engine's internal format, never exposed)
  • Call enginePool.execute() — direct in-JVM call, no HTTP
  • Map engine response via ResponseMapper → CalculateResponse JSON DTO
RestTemplate & SOAP are removed entirely No callTomcat2Async(), no JAXBUtil.parseRequest(soapXml), no RM3Util.wrapWithSoapEnvelope() in the external path. JAXB is used only for the internal RPSD engine protocol.

16. Verification Checklist

Engine Pool

  • 20 concurrent simulate calls complete successfully (no engine leaks)
  • Exception thrown inside simulate still returns engine to pool
  • 21st concurrent call returns error code 32 after ~3 seconds (not instantly)
  • MMED reload while 10 engines busy: reload waits, then succeeds
  • After reload, new rules are applied on next simulate

Invoice Calculation

  • Valid JSON request with known SKUs → HTTP 200 + correct JSON response
  • Response is application/json — no XML in response body
  • SKU not in price catalogue → listed in skippedSkus[], not silently dropped
  • Article in price map but not in OpenSearch → DB fallback used
  • Store not in OpenSearch → sentinel store 999999 used for price
  • NTot = sum of all line amounts (rounded to 3dp)
  • DTot = sum of all promotion discounts (rounded to 3dp)
  • ITot = NTot - DTot (rounded to 3dp)
  • SDi and CDi correctly split by promotionType < 10 / ≥ 10
  • SPr = NAm - SDi - CDi per line
  • SPI set for standard promotions; CPI set for coupons
  • Pre[] populated when extraResultInfo key 103 present

Security

  • POST /v1/invoice without X-API-Key → HTTP 401
  • POST /v1/invoice with wrong key → HTTP 401
  • POST /v1/invoice with correct key → HTTP 200
  • GET /actuator/health without key → HTTP 200 {"status":"UP"}
  • GET /swagger-ui.html without key → HTTP 200

MMED Operations

  • POST /api/v1/mmed/upload with XML < 200 bytes → error code 16
  • POST /api/v1/mmed/upload with invalid MMED → error code 14
  • POST /api/v1/mmed/upload with valid MMED → success, "not active yet"
  • POST /api/v1/mmed/reload → downloads from S3, reloads pool, returns timestamp
  • Requests during reload queue and succeed after reload completes

SOAP Compatibility

  • Request with <x:Envelope> prefix is parsed correctly
  • Request with <SOAP-ENV:Envelope> prefix is parsed correctly
  • Response includes xmlns="http://www.capgemini.com/rm3/pricing4.0" on Invoice tag
  • Response includes correct soapenv:encodingStyle attribute

17. Risks & Key Decisions

Risk / DecisionSeverityMitigation
RM3E JAR must be available in final Docker image MEDIUM Copy lib/RM3E-3.5.0.jar into image in Dockerfile. Do not rely on Maven repo for deployment.
RestHighLevelClient is deprecated in OpenSearch client >2.x MEDIUM Keep RestHighLevelClient for now (same as core service) — plan migration to OpenSearch Java client in a follow-up.
Articles silently dropped if price not found — callers may not notice MEDIUM Add a log.warn per dropped article with traceId. Consider adding a droppedArticles field to response in a future version.
MMED reload causes brief unavailability while engines are drained LOW ReadWriteLock ensures new requests queue rather than fail. Reload typically completes in <2s.
Context path change (/promotionservice/calculateinvoicecore) affects clients HIGH Configure new service with same context-path as core service in server.servlet.context-path. No client change needed.
X-API-Key is a new requirement — callers must be updated HIGH Coordinate with api-service team to add header before cutover. Alternatively, make auth opt-out initially.
Shadow engine feature in core service not ported LOW Explicitly out of scope. Shadow routing can be re-added as a separate feature flag in Phase 2 if needed.
Engine pool size (maxEngines=20) shared with all request types LOW MMED operations (upload/reload) are infrequent. Engine pool for calculate has exclusive access via ReadLock.
RPSD XML builder correctness — engine is a black box MEDIUM Port JAXB models exactly from core service. Validate by replaying real requests against both old and new service and diffing responses.
